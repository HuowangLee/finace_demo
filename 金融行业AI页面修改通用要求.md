# 金融行业 AI 页面修改通用要求

## 目标定位

页面应设计为业务人员实际使用的产品界面，而不是宣传页、方案汇报页、路演页或能力展示页。打开页面后，用户应能直接完成当前角色的核心工作，而不是阅读产品价值口号。

## 内容原则

1. 页面内容必须围绕真实业务工作流展开，例如检索、处理任务、查看待办、复核结果、更新资料、进入常用功能。
2. 不要在页面中出现解释性纠偏文案，例如“不是宣传功能”“不做营销包装”“这是实际使用页面”等。
3. 不要罗列多个无关项目或场景。每个页面只服务当前产品功能和当前业务角色。
4. 文案应像真实系统里的字段、提示、状态和操作说明，而不是销售材料或方案标题。
5. 指标要服务业务判断，例如待处理数量、解析状态、权限状态、资料更新时间、复核进度，而不是泛化的宣传指标。

## 页面结构要求

1. 顶部区域应提供当前页面名称、用户上下文、关键搜索或筛选入口。
2. 主内容区应展示当前用户最常处理的业务对象，例如任务列表、资料列表、问答结果、复核记录、最近访问。
3. 右侧或辅助区域可展示待办、状态概览、资料更新、风险提示、权限提醒等辅助信息。
4. 左侧导航应代表真实产品模块，并支持切换到对应内容。
5. 页面布局应保持后台产品的信息密度，避免大屏化、海报化和过度视觉包装。

## 交互要求

1. 左侧导航、标签页、筛选项、按钮等可点击元素必须有实际反馈。
2. 导航或标签页切换时，应至更新页面具体内容。
3. 搜索建议、快捷条件应能填充搜索框或改变列表内容。
4. 操作按钮可以用轻量提示模拟反馈，但不能只有提示而没有核心内容变化。
5. 如果页面是静态原型，也要通过前端状态切换模拟真实产品路径。

## 视觉要求

1. 使用企业后台产品风格，突出清晰、稳定、可操作。
2. 控制装饰性元素，不使用过强的宣传式渐变、大标题、口号和视觉噱头。
3. 卡片、表格、标签、状态徽标应服务信息组织，而不是单纯增加设计感。
4. 状态色应有业务含义，例如待复核、已入库、权限受控、解析失败、资料缺口。
5. 页面应优先保证可读性、可扫描性和操作路径清晰。

## 技术实现要求

1. 每个重做页面应尽量独立实现，不依赖公共模板渲染逻辑。
2. 不使用公共配置对象生成页面，例如统一 `PROTOTYPE_CONFIG` 式渲染。
3. 如需保留静态原型，应在当前 HTML 内完成必要样式和交互逻辑。
4. 避免把多个页面抽成同一套展示模板，除非该模板本身就是产品真实共用组件。
5. 修改完成后检查页面中是否残留宣传、演示、虚构项目、PR 等不符合实际产品首页的表述。

## 检查清单

- 页面打开后，用户是否知道下一步该做什么？
- 页面里的信息是否像真实业务系统，而不是方案 PPT？
- 每个主要按钮、标签页、导航是否有对应反馈？
- 是否删除了“解释为什么这样设计”的界面文案？
- 是否避免了无关项目堆砌和泛化能力展示？
- 是否保留了当前业务角色真正需要的任务、资料、状态和操作？
- 是否没有依赖公共模板或公共渲染逻辑？
